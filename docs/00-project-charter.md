# SoloOps 项目章程

## 1. 项目愿景

SoloOps 是一个面向独立开发者、小团队和早期创业项目的 AI 应用运维副驾。它把云资源、监控指标、部署状态、日志线索和运维规则组织成可验证证据，帮助用户完成“巡检发现 -> 风险解释 -> 修复计划 -> 人工审批 -> 受控执行 -> 验证审计”的闭环。

项目的核心价值不是让大模型直接登录服务器执行命令，而是把运维决策链条做得更快、更透明、更安全：每个结论有证据，每个修复有边界，每次变更可审批、可回滚、可追踪。

## 2. 目标用户

| 用户 | 痛点 | SoloOps 首版收益 |
| --- | --- | --- |
| 独立开发者 | 项目多、告警多、缺少系统化巡检 | 每日自动生成风险清单和修复建议 |
| 小型 SaaS 团队 | 无专职 SRE，线上问题依赖经验排查 | 形成标准化诊断和审批流程 |
| 初创团队后端负责人 | 云权限、变更记录和安全边界混乱 | 最小权限接入、变更留痕、审计可复盘 |
| 求职项目展示者 | 普通 CRUD 项目区分度低 | 展示云原生、Agent、安全、可观测和工程治理能力 |

## 3. 问题陈述

小团队的线上应用通常由少量 ECS、数据库、对象存储、缓存和 Docker/Compose 组成。问题并不总是来自复杂架构，而是来自持续被忽略的风险：公网数据库端口、磁盘空间耗尽、容器反复重启、备份失效、证书过期、成本异常、告警无人闭环。

传统监控工具能发出告警，但往往缺少上下文解释和修复计划；通用 Agent 可以生成建议，但如果没有权限边界和审计，很难进入真实生产环境。SoloOps 的定位是在两者之间提供一个可信运维控制面。

## 4. 产品原则

1. **证据优先**：所有 Finding 必须关联资源、指标、日志或配置快照。
2. **默认只读**：巡检、诊断和计划阶段不允许修改云资源。
3. **人工最终审批**：任何有副作用的动作必须经过人类批准。
4. **受限 Playbook**：执行器只接受白名单动作，不提供任意 Shell、SSH 或模型自由拼接 API。
5. **最小权限**：读角色和写角色分离，写权限短时、单资源、单动作。
6. **可审计和可回放**：保留计划、审批、执行、验证和 Trace。
7. **先确定性后智能化**：规则和状态机是底座，LLM 只用于解释、归纳和辅助计划。

## 5. MVP 成功标准

- 在无真实云密钥条件下，Mock 场景能完整演示扫描、Finding、计划、审批、执行和审计。
- 接入只读云角色后，能读取 ECS、安全组、CloudMonitor 指标，并生成至少 5 类风险 Finding。
- 未审批的执行请求 100% 被拒绝。
- 未命中白名单的动作 100% 被拒绝。
- 每次扫描、计划、审批和执行都能关联 `trace_id`、操作者、资源、动作和结果。
- 项目具备一键本地启动、自动化测试、OpenAPI 文档、部署手册和演示脚本。

## 6. 范围边界

### In scope

- 云资源只读盘点：ECS、RDS、OSS、Redis/Tair、安全组、负载均衡。
- 指标巡检：磁盘、CPU、内存、网络、容器重启、错误率。
- 安全巡检：公网暴露端口、弱安全组、公开 Bucket、过期证书。
- 可靠性巡检：备份状态、实例健康、部署版本、最近变更。
- Finding 生命周期、修复计划、审批、执行、验证和审计。
- 受控 Playbook：撤销指定风险安全组规则、触发备份、采集诊断包。
- Web Console、API、Worker、数据库持久化、可观测性和评测集。

### Out of scope

- 任意 SSH、任意 Shell、模型直接修改服务器。
- 自动绕过云平台权限、安全组或审批。
- 自研监控系统替代 Prometheus/SLS/CloudMonitor。
- 高风险破坏性动作自动执行，例如删除实例、清空数据库、重装系统。
- 多云全量覆盖；MVP 优先阿里云，其他云通过 Provider 接口扩展。

## 7. 秋招项目叙事

SoloOps 适合作为企业级后端/AI 工程项目展示，重点讲清：

- 云原生系统设计：API、Worker、PostgreSQL、Redis、OSS、Docker、OpenAPI。
- Agent 工程化：结构化工具、状态机、Trace、评测、成本和失败兜底。
- 安全工程：RAM/STS、最小权限、审批、审计、白名单执行。
- 可观测性：请求链路、任务状态、Finding 证据、执行验证。
- 工程质量：测试金字塔、CI、ADR、部署运行手册。

## 8. 主要风险

| 风险 | 后果 | 缓解 |
| --- | --- | --- |
| 云权限配置过大 | 项目被质疑不可安全上线 | 读写角色分离、会话策略、权限文档和拒绝任意执行 |
| LLM 输出不可控 | 错误修复建议或误导用户 | 结构化 JSON、规则校验、人工审批、不可直接执行 |
| 接入真实云 API 工作量过大 | MVP 迟迟不可演示 | Mock Provider 先闭环，真实 Provider 分阶段接入 |
| 只是监控面板 | 项目差异化不足 | 强调 Finding 到审批执行的闭环和审计链 |
| 单人开发范围失控 | 无法按期完成 | 严格 8 周路线图，先完成核心链路再扩展 UI/Agent |
