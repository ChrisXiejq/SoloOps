# SoloOps 项目章程

## 1. 项目愿景

SoloOps 是一个面向独立开发者、小团队和早期创业项目的 AI 运维治理层。它不重复建设阿里云 CloudMonitor、ARMS、SLS、ECS 健康检查和 OOS 已经提供的监控与自动化能力，而是把这些原生信号、资源配置、部署上下文和团队策略组织成可验证证据，帮助用户完成“告警归因 -> 风险解释 -> 修复计划 -> 人工审批 -> 受控执行 -> 验证审计”的闭环。

项目的核心价值不是再造一套云监控，也不是让大模型直接登录服务器执行命令，而是把“告警之后怎么办”这条运维决策链做得更快、更透明、更安全：每个结论有证据，每个修复有边界，每次变更可审批、可回滚、可追踪。

## 2. 目标用户

| 用户 | 痛点 | SoloOps 首版收益 |
| --- | --- | --- |
| 独立开发者 | 阿里云控制台分散、告警来了不知道下一步 | 把告警聚合成带证据的 Finding 和修复建议 |
| 小型 SaaS 团队 | 无专职 SRE，CloudMonitor/OOS 会用但缺少治理流程 | 形成标准化诊断、审批、执行和复盘流程 |
| 初创团队后端负责人 | 云权限、OOS 模板、变更记录和安全边界混乱 | 最小权限接入、变更留痕、审计可复盘 |
| 求职项目展示者 | 普通 CRUD 项目区分度低 | 展示云原生、Agent、安全、可观测和工程治理能力 |

## 3. 问题陈述

小团队的线上应用通常由少量 ECS、数据库、对象存储、缓存和 Docker/Compose 组成。阿里云已经能很好地发现很多底层问题：CloudMonitor/ARMS/SLS 提供指标、链路和日志，ECS 有实例健康状态，OOS 可以编排自动化任务。但真实痛点经常出现在告警之后：这条告警是否重要、证据是否充分、是否和最近发布有关、应该用哪个模板修、谁来批准、执行后是否真的恢复、事后如何复盘。

SoloOps 的定位是在阿里云原生运维生态之上提供一个可信治理控制面：不替代监控采集和 OOS 编排，而是把原生能力串成面向团队的 AI 决策闭环。

## 4. 产品原则

1. **原生优先**：优先接入 CloudMonitor、ARMS、SLS、ECS 健康检查、OOS 和 ActionTrail，不重复造基础监控轮子。
2. **证据优先**：所有 Finding 必须关联告警、健康事件、资源、指标、日志、配置快照或部署上下文。
3. **默认只读**：归因、诊断和计划阶段不允许修改云资源。
4. **人工最终审批**：任何有副作用的动作必须经过人类批准。
5. **受限 Playbook**：执行器只接受白名单动作，可调用 OOS 模板，但不提供任意 Shell、SSH 或模型自由拼接 API。
6. **最小权限**：读角色和写角色分离，写权限短时、单资源、单动作。
7. **可审计和可回放**：保留告警来源、计划、审批、执行、验证和 Trace。
8. **先确定性后智能化**：规则和状态机是底座，LLM 只用于解释、归纳和辅助计划。

## 5. MVP 成功标准

- 在无真实云密钥条件下，Mock 场景能完整演示扫描、Finding、计划、审批、执行和审计。
- 接入只读云角色后，能读取 ECS、安全组、CloudMonitor 健康/指标信号，并将至少 5 类原生信号或配置风险归并为 Finding。
- 未审批的执行请求 100% 被拒绝。
- 未命中白名单的动作 100% 被拒绝。
- 每次扫描、计划、审批和执行都能关联 `trace_id`、操作者、资源、动作和结果。
- 项目具备一键本地启动、自动化测试、OpenAPI 文档、部署手册和演示脚本。

## 6. 范围边界

### In scope

- 阿里云原生信号接入：CloudMonitor 告警/指标、ECS 健康状态、SLS 日志摘要、ARMS 应用事件、OOS 执行结果。
- 云资源只读盘点：ECS、RDS、OSS、Redis/Tair、安全组、负载均衡。
- 证据归并：把告警、指标、配置、部署版本和团队策略归并为 Finding。
- 安全巡检：公网暴露端口、弱安全组、公开 Bucket、过期证书。
- 可靠性巡检：备份状态、实例健康、部署版本、最近变更。
- Finding 生命周期、修复计划、审批、执行、验证和审计。
- 受控 Playbook：撤销指定风险安全组规则、触发备份、采集诊断包。
- Web Console、API、Worker、数据库持久化、可观测性和评测集。

### Out of scope

- 任意 SSH、任意 Shell、模型直接修改服务器。
- 自动绕过云平台权限、安全组或审批。
- 自研监控系统替代 Prometheus/SLS/CloudMonitor/ARMS。
- 自研 OOS 替代品或通用任务编排平台。
- 高风险破坏性动作自动执行，例如删除实例、清空数据库、重装系统。
- 多云全量覆盖；MVP 优先阿里云，其他云通过 Provider 接口扩展。

## 7. 秋招项目叙事

SoloOps 适合作为企业级后端/AI 工程项目展示，重点讲清：

- 云原生系统设计：API、Worker、PostgreSQL、Redis、OSS、Docker、OpenAPI。
- Agent 工程化：结构化工具、状态机、Trace、评测、成本和失败兜底。
- 安全工程：RAM/STS、最小权限、审批、审计、白名单执行。
- 可观测性：请求链路、任务状态、Finding 证据、执行验证。
- 工程质量：测试金字塔、CI、ADR、部署运行手册。

## 8. 主要风险

| 风险 | 后果 | 缓解 |
| --- | --- | --- |
| 和阿里云原生能力重复 | 项目价值被质疑 | 明确定位为治理层，优先接入 CloudMonitor/ARMS/SLS/OOS |
| 云权限配置过大 | 项目被质疑不可安全上线 | 读写角色分离、会话策略、权限文档和拒绝任意执行 |
| LLM 输出不可控 | 错误修复建议或误导用户 | 结构化 JSON、规则校验、人工审批、不可直接执行 |
| 接入真实云 API 工作量过大 | MVP 迟迟不可演示 | Mock Provider 先闭环，真实 Provider 分阶段接入 |
| 只是监控面板 | 项目差异化不足 | 强调 Finding 到审批执行的闭环和审计链 |
| 单人开发范围失控 | 无法按期完成 | 严格 8 周路线图，先完成核心链路再扩展 UI/Agent |
