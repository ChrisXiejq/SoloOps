# PRD：SoloOps AI 应用运维副驾

**版本**：v0.1  
**状态**：开发前基线  
**目标**：完成可演示、可部署、可审计的企业级运维 Agent MVP

## 1. 用户故事

### US-01：配置工作区与云账号

作为团队负责人，我希望在工作区中配置云账号、环境、资源标签和权限状态，使系统只巡检我授权的资源。

**验收**：用户可创建工作区；配置 Provider 为 `mock` 或 `aliyun-readonly`；系统展示权限检查结果；缺少只读权限时不可启动真实扫描。

### US-02：执行证据化巡检

作为运维负责人，我希望一键扫描云资源和运行状态，得到带证据的风险清单。

**验收**：扫描任务返回 `scan_id`；每个 Finding 包含规则 ID、严重级别、资源 ID、描述、证据来源、观测时间和建议动作。

### US-03：查看风险解释

作为开发者，我希望理解为什么某个资源有风险，以及它可能造成什么影响。

**验收**：Finding 详情页展示证据、规则说明、影响范围、误报提示、修复选项和历史相同问题。

### US-04：生成修复计划

作为负责人，我希望系统根据 Finding 生成受限的修复计划，而不是直接执行修改。

**验收**：计划只允许来自白名单的 `remediation_action`；计划包含目标资源、预期影响、回滚方式、验证步骤和风险等级。

### US-05：人工审批与执行

作为审批人，我希望在看到证据和影响后批准或拒绝修复计划。

**验收**：未审批计划不可执行；审批记录包含审批人、意见和时间；执行器记录执行前后证据。

### US-06：审计和复盘

作为团队负责人，我希望按资源、规则、人员和时间查看历史巡检和变更，便于复盘和面试展示。

**验收**：系统可查询扫描历史、Finding 生命周期、审批记录、执行结果、验证结论和 Trace 摘要。

## 2. 核心流程

1. 用户创建工作区并配置云 Provider。
2. 系统执行权限预检，确认只读角色可用。
3. 用户触发扫描，Worker 拉取资源、指标、配置和运行状态。
4. Rule Engine 生成 Finding，并保存证据快照。
5. 用户查看 Finding 详情，选择生成修复计划。
6. Planner 根据白名单 Playbook 生成计划。
7. 审批人批准或拒绝计划。
8. Executor 使用短时写角色执行受控动作，或在本地 dry run。
9. Verifier 重新读取资源和指标，判断修复是否生效。
10. Audit Service 固化全链路记录。

## 3. 功能需求

### 3.1 工作区与权限

- 支持单租户 MVP，数据模型预留 `tenant_id`。
- 角色：Owner、Operator、Approver、Viewer。
- Owner 管理 Provider、成员和执行开关。
- Operator 可扫描和创建计划，不可审批自己的高风险计划。
- Approver 可审批和拒绝计划，不直接修改执行参数。

### 3.2 Provider 接入

- MVP 默认支持 `MockCloudProvider`，保证离线演示。
- 阿里云 Provider 分为只读 Provider 和写执行 Adapter。
- Provider 接口必须区分读取资源、读取指标、读取配置和执行动作。
- Provider 错误需要归一化为可展示的连接、权限、限流、资源不存在等类型。

### 3.3 扫描与规则

首批规则：

| 规则 ID | 类型 | 严重级别 | 说明 |
| --- | --- | --- | --- |
| SG-001 | 安全 | Critical | PostgreSQL 5432 对公网开放 |
| ECS-001 | 可靠性 | High | 实例磁盘使用率超过阈值 |
| ECS-002 | 可靠性 | High | 容器重启次数异常 |
| RDS-001 | 可靠性 | High | 最近备份过期或不存在 |
| OSS-001 | 安全 | High | Bucket 公共读写风险 |
| TLS-001 | 可用性 | Medium | 证书即将过期 |
| COST-001 | 成本 | Medium | 单日成本异常上涨 |

### 3.4 Finding 管理

- 状态：`OPEN / ACKED / PLANNED / APPROVED / EXECUTING / RESOLVED / WONT_FIX / FALSE_POSITIVE`。
- 支持按严重级别、资源、规则、状态和环境筛选。
- 支持误报标记，但必须填写原因。
- 支持相同资源相同规则的去重和重新打开。

### 3.5 修复计划与 Playbook

- 修复计划由确定性 Playbook 生成，LLM 只能补充解释和风险提示。
- 每个 Playbook 定义输入 Schema、权限、执行步骤、验证步骤和回滚步骤。
- 首批 Playbook：
  - `revoke_public_postgres_rule`：撤销精确匹配的公网 PostgreSQL 安全组规则。
  - `collect_disk_diagnosis`：采集磁盘占用证据，不修改资源。
  - `collect_container_diagnosis`：采集容器状态和日志元数据。
  - `trigger_rds_backup`：触发指定 RDS 实例备份并验证备份任务。

### 3.6 审批与执行

- 所有写动作默认需要审批。
- 执行前校验：计划存在、审批存在、动作白名单、资源匹配、权限有效、幂等键有效。
- 执行后必须自动验证；验证失败时进入 `FAILED`，并展示回滚建议。
- dry run 模式必须可演示完整链路。

### 3.7 Web Console

```text
登录/本地演示入口
└── 工作区
    ├── 首页：风险概览、最新扫描、待审批
    ├── 扫描任务：触发扫描、查看进度和历史
    ├── Findings：风险列表、筛选、详情
    ├── Remediation：计划、审批、执行和验证
    ├── 资源视图：实例、安全组、数据库、存储、服务
    ├── Playbooks：白名单动作和风险说明
    ├── Audit：审计日志、Trace、导出
    └── 设置：Provider、成员、执行开关、阈值
```

## 4. 非功能需求

| 项目 | MVP 指标 |
| --- | --- |
| 可用性 | 本地和测试环境 API 99% 可用 |
| 响应时间 | 普通查询 P95 < 300ms；扫描异步化 |
| 扫描规模 | MVP 支持 100 台实例、500 条安全组规则 |
| 安全 | 无审批写动作成功率为 0 |
| 审计 | 100% 计划、审批、执行有审计记录 |
| 可测试性 | 核心领域规则单元测试覆盖；API 主链路集成测试 |
| 可部署性 | 一条命令本地启动；生产有 Docker Compose 手册 |

## 5. 明确不做

- 不实现通用远程命令执行平台。
- 不让模型生成未校验的云 API 参数并直接执行。
- 不在 MVP 中追求复杂 Kubernetes 集群运维。
- 不支持绕过云平台 IAM、审计和风控。
- 不把“建议”伪装成“已验证事实”。
