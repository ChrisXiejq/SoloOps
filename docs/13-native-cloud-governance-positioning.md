# 阿里云原生生态定位

## 1. 核心结论

SoloOps 不替代阿里云 CloudMonitor、ARMS、SLS、ECS 健康检查、ActionTrail 或 OOS。它的定位是这些原生能力之上的 AI 运维治理层：

```text
阿里云原生信号与自动化
  -> 告警/健康事件/日志摘要/执行记录/资源配置
  -> SoloOps 证据归并和风险解释
  -> 修复计划和人工审批
  -> 受控 Playbook / OOS 模板执行
  -> 执行后验证和审计复盘
```

如果用户只是需要基础监控、指标阈值、告警通知或常见云上自动化，应优先使用阿里云原生产品。SoloOps 解决的是“告警之后如何治理”的问题。

## 2. 阿里云原生能力边界

| 阿里云能力 | 强项 | SoloOps 不重复做 |
| --- | --- | --- |
| CloudMonitor | 指标、告警、实体健康度、阈值规则 | 不自建通用指标采集和告警平台 |
| ARMS | 应用链路、调用、错误、应用侧观测 | 不自研 APM |
| SLS | 日志采集、查询、告警、投递 | 不自研日志平台 |
| ECS 健康检查 | 实例状态、系统状态、性能风险事件 | 不替代底层实例健康判断 |
| OOS | 运维模板、定时/告警/事件触发执行、记录投递 | 不自研通用编排引擎 |
| ActionTrail | 云 API 操作审计 | 不自研云账号审计源 |
| RAM/STS | 权限和临时凭证 | 不绕过云权限体系 |

## 3. SoloOps 的增量价值

### 3.1 告警归并

一个线上问题可能同时触发 CloudMonitor 磁盘告警、SLS 错误日志、ECS 状态异常和最近发布记录。SoloOps 将这些原生信号归并为一个 Finding，减少重复告警和碎片化排查。

### 3.2 业务上下文补全

阿里云知道资源状态，但不一定知道团队的部署清单、服务拓扑、代码版本、Runbook、审批策略和业务优先级。SoloOps 把这些上下文纳入证据链。

### 3.3 安全执行治理

OOS 能执行模板，SoloOps 负责决定一个模板是否可以被当前 Finding 使用：

- Finding 证据是否充分。
- 模板是否在 Playbook Registry 中登记。
- 参数是否和证据匹配。
- 是否需要审批。
- 审批是否过期。
- 执行后如何验证。

### 3.4 AI 解释和复盘

LLM 不直接执行命令，而是基于证据生成风险解释、审批摘要、复盘报告和下一步建议。这样既能利用模型能力，又不把生产变更交给自由文本。

### 3.5 面向小团队的一站式工作台

小团队常见问题不是没有云产品，而是控制台分散、流程不成体系。SoloOps 提供统一视图：

- 今天有哪些高风险 Finding。
- 每个 Finding 证据是什么。
- 推荐哪个 Playbook 或 OOS 模板。
- 谁批准了。
- 执行是否成功。
- 后续如何预防。

## 4. 产品边界

### 必须做

- 接入 CloudMonitor 告警、指标和实体健康状态。
- 接入 ECS/RDS/OSS 等资源配置。
- 接入 OOS 模板列表、执行状态和执行记录。
- 保存 Finding、计划、审批、执行、验证和审计链。
- 用 RAM/STS 和 Playbook Registry 限制执行边界。

### 不应该做

- 自研完整监控、日志、链路追踪或告警平台。
- 自研 OOS 替代品。
- 让 LLM 直接拼接云 API 参数并执行。
- 让用户通过 SoloOps 绕过 RAM、ActionTrail 或 OOS 审计。
- 对所有云资源做大而全的多云管理。

## 5. 面试叙事

当被问到“为什么不用阿里云原生能力”时，回答应是：

> 生产运维当然优先使用阿里云原生能力。SoloOps 不是替代 CloudMonitor、ARMS、SLS 或 OOS，而是把这些能力上升为一个可控的 AI 运维治理系统。它解决的是告警之后的归因、解释、审批、安全执行、验证和复盘问题，并展示了云原生集成、权限治理、Agent 安全边界和工程化评测能力。

## 6. 实施优先级

1. 保留 Mock Provider，作为离线演示和测试基线。
2. 接入 CloudMonitor 告警/指标读取。
3. 接入 ECS 资源、安全组和健康状态。
4. 接入 OOS 模板元数据和执行记录，只读展示。
5. 将 Playbook 与 OOS 模板建立映射。
6. 审批后调用 OOS 模板执行低风险动作。
7. 用 ActionTrail 和 OOS 执行记录补齐审计链。
